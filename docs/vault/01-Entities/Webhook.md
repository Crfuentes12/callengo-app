# Webhook

Outbound webhooks that Callengo sends to customer-configured HTTPS endpoints when events occur in the platform. This system enables customers to integrate Callengo with automation tools like Zapier, Make (Integromat), and n8n, as well as custom backend systems. Each webhook delivery is signed with HMAC-SHA256 for payload verification, and endpoints are automatically disabled after consecutive delivery failures.

This entity covers the **outbound** webhook system (Callengo -> customer endpoints). For **inbound** webhooks (Stripe -> Callengo), see [[Stripe Integration]]. For inbound Bland AI webhooks, see [[Bland AI]].

## Database Tables

### `webhook_endpoints`

Stores the customer-configured HTTPS URLs that receive webhook events. Each endpoint subscribes to a set of event types and has its own HMAC signing secret.

Created in migration `20260302_outbound_webhooks.sql`.

| `company_id` | `UUID` | -- | NOT NULL | Foreign key to `companies(id)` with `ON DELETE CASCADE`. Scopes the endpoint to a [[Company]]. |

### `webhook_deliveries`

Immutable log of every webhook delivery attempt. Each row represents a single HTTP POST to a webhook endpoint. Used for debugging, retry analysis, and delivery status reporting in the UI.

Created in migration `20260302_outbound_webhooks.sql`.

#### Full Column Specification

| Column | Type | Default | Nullable | Description |
|--------|------|---------|----------|-------------|
| `id` | `UUID` | `gen_random_uuid()` | NOT NULL | Primary key. |
| `endpoint_id` | `UUID` | -- | NOT NULL | Foreign key to `webhook_endpoints(id)` with `ON DELETE CASCADE`. Links the delivery to its target endpoint. When an endpoint is deleted, its entire delivery history is cascade-deleted. |
| `company_id` | `UUID` | -- | NOT NULL | Foreign key to `companies(id)` with `ON DELETE CASCADE`. Denormalized from `webhook_endpoints.company_id` for RLS efficiency and direct company-scoped queries without joining. |
| `event_type` | `TEXT` | -- | NOT NULL | The event type string (e.g., `'call.completed'`). Stored for filtering and analysis. |
| `event_id` | `TEXT` | -- | NOT NULL | Unique event identifier generated by the dispatcher: `evt_` followed by 24 hex characters (12 random bytes). Sent in the `X-Callengo-Delivery` header. Recipients can use this for idempotency -- if they receive the same `event_id` twice (e.g., due to a retry or network duplicate), they can safely skip the second delivery. |
| `payload` | `JSONB` | -- | NOT NULL | The complete JSON payload that was sent in the HTTP POST body. Structure: `{ id: event_id, type: event_type, created_at: ISO timestamp, data: { ...event-specific fields } }`. |
| `status` | `TEXT` | `'pending'` | NOT NULL | Delivery outcome: `'pending'` (in progress or not yet attempted), `'success'` (HTTP 2xx response), or `'failed'` (non-2xx response, timeout, or network error). |
| `http_status` | `INTEGER` | `NULL` | YES | The HTTP status code returned by the remote server. `NULL` if the request never completed (timeout, DNS failure, connection refused). |
| `response_body` | `TEXT` | `NULL` | YES | The first 1,024 bytes (1KB) of the response body from the remote server. Truncated to prevent large responses from consuming storage. Useful for debugging failed deliveries. |
| `error_message` | `TEXT` | `NULL` | YES | Human-readable error description for failed deliveries. Examples: `"HTTP 502"`, `"Request timed out"`, `"Connection refused"`. `NULL` for successful deliveries. |
| `attempt_number` | `INTEGER` | `1` | NOT NULL | The delivery attempt number. Currently always `1` as the system does not implement automatic retries. Future retry logic would increment this counter for subsequent attempts. |
| `delivered_at` | `TIMESTAMPTZ` | `NULL` | YES | Timestamp when the delivery completed (success or failure). `NULL` for `pending` deliveries. |
| `duration_ms` | `INTEGER` | `NULL` | YES | Total time in milliseconds from request initiation to response completion (or timeout). Useful for monitoring endpoint latency and identifying slow webhooks. |
| `created_at` | `TIMESTAMPTZ` | `NOW()` | NOT NULL | Row creation timestamp. Deliveries older than 30 days can be purged via cron (cleanup logic noted in migration but not yet implemented as a scheduled job). |

## HMAC-SHA256 Signature Verification

Every outbound webhook delivery includes cryptographic signing headers that allow recipients to verify the payload was genuinely sent by Callengo and has not been tampered with.

### Signing Process (Callengo side)

1. The payload is serialized to JSON string.
2. An HMAC-SHA256 signature is computed: `HMAC-SHA256(payload_json_string, endpoint_secret)`.
3. The signature and a Unix timestamp are included in HTTP headers.

### HTTP Headers Sent

| Header | Value | Purpose |
|--------|-------|---------|
| `Content-Type` | `application/json` | Standard JSON content type. |
| `X-Callengo-Signature` | Hex-encoded HMAC-SHA256 | The cryptographic signature of the payload body. Recipients compute the same HMAC with their copy of the secret and compare. |
| `X-Callengo-Timestamp` | Unix timestamp (seconds) | When the request was initiated. Recipients should reject payloads with timestamps older than 5 minutes to prevent replay attacks. |
| `X-Callengo-Event` | Event type string | The event type (e.g., `call.completed`). Allows recipients to route events without parsing the body. |
| `X-Callengo-Delivery` | Event ID string | Unique delivery identifier for idempotency. Format: `evt_` + 24 hex chars. |
| `User-Agent` | `Callengo-Webhooks/1.0` | Identifies the sender as Callengo's webhook system. |

### Verification Process (Recipient side)

The `verifyInboundWebhookSignature()` function in `src/lib/webhooks.ts` provides a reference implementation:

1. Extract the `X-Callengo-Signature` and `X-Callengo-Timestamp` headers.
2. Compute `HMAC-SHA256(raw_request_body, webhook_secret)` in hex.
3. Compare using `crypto.timingSafeEqual()` (constant-time comparison to prevent timing attacks).
4. Check that `X-Callengo-Timestamp` is within 300 seconds (5 minutes) of the current time to prevent replay attacks.

### Secret Generation

Secrets are generated by `generateWebhookSecret()`:
```
whsec_ + crypto.randomBytes(24).toString('hex')
```
Resulting format: `whsec_` followed by 48 hexadecimal characters (192 bits of entropy).

## Dispatch Pipeline

The dispatch pipeline is implemented in `src/lib/webhooks.ts` and is invoked throughout the application wherever events occur.

### `dispatchWebhookEvent(companyId, eventType, data)`

This is the main entry point, called fire-and-forget (errors are logged but never thrown to the caller).

1. **Endpoint discovery:** Queries `webhook_endpoints` for rows matching `company_id`, `is_active = true`, `auto_disabled_at IS NULL`, and `events @> [eventType]` (array contains).
2. **Event ID generation:** Creates a unique event identifier: `evt_` + 24 random hex chars.
3. **Payload construction:** Wraps the event data in a standard envelope: `{ id, type, created_at, data }`.
4. **Concurrent delivery:** Uses `Promise.allSettled()` to deliver to all matching endpoints in parallel, ensuring one slow or failing endpoint does not block others.

### `deliverToEndpoint(endpoint, eventType, eventId, payload)`

Handles a single delivery attempt:

1. **Signing:** Computes HMAC-SHA256 of the JSON payload string using the endpoint's secret.
2. **HTTP POST:** Sends the request with a 10-second timeout (`DELIVERY_TIMEOUT_MS = 10000`) using `AbortController`.
3. **Response capture:** Reads the first 1KB of the response body. Records HTTP status code.
4. **Success determination:** HTTP 2xx = success; anything else = failure. Timeouts and network errors are also failures.
5. **Delivery logging:** Inserts a row into `webhook_deliveries` with all delivery metadata.
6. **Health update (success):** Updates `last_triggered_at`, `last_success_at`, resets `consecutive_failures` to 0.
7. **Health update (failure):** Increments `consecutive_failures`, updates `last_failure_at` and `last_failure_reason`. If `consecutive_failures >= MAX_CONSECUTIVE_FAILURES` (10), sets `auto_disabled_at = NOW()` and `is_active = false`.

### Auto-Disable and Re-Enable

When an endpoint accumulates 10 consecutive delivery failures, it is automatically disabled:
- `is_active` is set to `false`
- `auto_disabled_at` is set to the current timestamp
- The dispatcher skips this endpoint for all future events

When a user manually re-enables the endpoint (via `updateWebhookEndpoint()` with `is_active: true`):
- `auto_disabled_at` is reset to `NULL`
- `consecutive_failures` is reset to `0`
- The endpoint resumes receiving events immediately

### Test Webhooks

The `sendTestWebhook()` function sends a test payload to a specific endpoint:
- Event type: `'test'`
- Event ID: `evt_test_` + 16 random hex chars
- Payload data: `{ message: "This is a test webhook from Callengo", endpoint_id }`
- Returns `{ success, httpStatus, error, durationMs }` for immediate feedback in the UI
- Does **not** record a delivery log row (test deliveries are ephemeral)

## CRUD Operations

All CRUD functions in `src/lib/webhooks.ts` use the `supabaseAdmin` (service role) client.

| Function | Parameters | Description |
|----------|-----------|-------------|
| `listWebhookEndpoints(companyId)` | Company ID | Returns all endpoints ordered by `created_at DESC`. |
| `createWebhookEndpoint(companyId, url, events, description?)` | Company ID, URL, event types array, optional description | Validates URL via `validateWebhookUrl()`, generates secret via `generateWebhookSecret()`, inserts row. | `updateWebhookEndpoint(endpointId, companyId, updates)` | Endpoint ID, Company ID, partial update object | Updates URL, events, description, or active status. Re-validates URL if changed. Resets failure tracking when re-enabling. |

## URL Validation (SSRF Prevention)

The `validateWebhookUrl()` function prevents Server-Side Request Forgery (SSRF) by blocking URLs that could cause Callengo's server to make requests to internal infrastructure.

### Blocked Patterns

| Category | Blocked Values | Error Message |
|----------|---------------|---------------|
| Protocol | Anything except `https:` | "Webhook URL must use HTTPS" |
| Localhost | `localhost`, `127.0.0.1`, `::1`, `0.0.0.0` | "Webhook URL cannot point to localhost" |
| Private IP: Class A | `10.x.x.x` | "...private IP range (10.x.x.x)" |
| Private IP: Class B | `172.16.x.x` through `172.31.x.x` | "...private IP range (172.16-31.x.x)" |
| Private IP: Class C | `192.168.x.x` | "...private IP range (192.168.x.x)" |
| Link-local | `169.254.x.x` | "...link-local range (169.254.x.x)" |
| Internal hostnames | Ending in `.internal`, `.local`, `.localhost`, `.svc.cluster` | "...internal hostnames" |

## Integration with Zapier / Make / n8n

Webhooks are available to all plans from Starter and above (Free/Starter/Growth include webhooks in their feature set). Customers configure their automation tool to receive Callengo webhooks:

1. **Zapier:** Create a "Webhooks by Zapier" trigger (Catch Hook). Copy the Zapier URL into Callengo's webhook endpoint configuration. Use the signing secret for optional verification in a Zapier code step.
2. **Make (Integromat):** Create a "Webhooks" module (Custom Webhook). Similar flow to Zapier.
3. **n8n:** Create a "Webhook" node. n8n supports custom header-based verification matching Callengo's `X-Callengo-Signature` format.
