# Company

The Company is the top-level tenant entity in Callengo and the root of the entire multi-tenant data hierarchy. Every single piece of data in the platform -- contacts, agents, campaigns, calls, subscriptions, billing events, calendar events, integrations, team invitations, voicemails, follow-ups, notifications, and webhooks -- is scoped to a Company through `company_id` foreign keys enforced by [[RLS Patterns|Row Level Security]]. A Company represents a single paying customer organization on the platform: a business that uses Callengo's AI voice agents to make outbound calls.

When a new user signs up for Callengo, the [[Onboarding Flow]] creates a Company record and a corresponding `company_settings` row in the same transaction. The user who creates the Company is automatically assigned the `owner` role (see [[User]]). From that point forward, every entity the user creates -- every [[Contact]] imported, every [[Agent]] configured, every [[Campaign]] launched -- is permanently bound to that Company's `company_id`.

The Company entity is implemented across two tightly coupled database tables: `companies` (the core identity) and `company_settings` (operational configuration). These exist in a strict 1:1 relationship enforced by `company_settings.company_id` being both a foreign key and the effective primary key of the settings table.

## Database Table: `companies`

The `companies` table holds the core identity and metadata for each tenant organization.

| Column | Type | Default | Nullable | Description |
|--------|------|---------|----------|-------------|
| `id` | `UUID` | `gen_random_uuid()` | NOT NULL | Primary key. Generated automatically on insert. Referenced by every other table in the system via `company_id` foreign keys. |
| `name` | `TEXT` | — | NOT NULL | The display name of the company, shown in the dashboard header, team management, and admin Command Center. Set during onboarding. |
| `website` | `TEXT` | — | YES | The company's website URL. Collected during onboarding. Used by the AI agents to provide context about the company when making calls (e.g., "I'm calling from {{company_name}}, we're a..."). |
| `description` | `TEXT` | — | YES | A free-text description of the company. Can be provided during onboarding or edited in settings. Used to enrich agent prompts with company context. |
| `logo_url` | `TEXT` | — | YES | URL to the company's logo image. Displayed in the dashboard sidebar, team invitations, and public-facing elements. Stored as a Supabase Storage URL or external URL. |
| `favicon_url` | `TEXT` | — | YES | URL to the company's favicon. Used in browser tab when white-labeling is applied (Enterprise plan). |
| `industry` | `TEXT` | — | YES | The company's industry vertical (e.g., "Healthcare", "Real Estate", "Legal", "SaaS"). Used for analytics segmentation in the [[Command Center]] and to tailor agent behavior. |
| `context_data` | `JSONB` | — | YES | Structured data extracted from the company's website via automated scraping. Contains information like products/services offered, team size, target market, and key value propositions. This data is fed to AI agents to make their conversations more contextually relevant. |
| `context_summary` | `TEXT` | — | YES | A natural-language summary of `context_data`, generated by [[OpenAI]] GPT-4o-mini. This summary is injected directly into agent system prompts so agents can speak knowledgeably about the company. |
| `context_extracted_at` | `TIMESTAMPTZ` | — | YES | Timestamp of when `context_data` and `context_summary` were last extracted/generated. Used to determine staleness and trigger re-extraction. |
| `deleted_at` | `TIMESTAMPTZ` | `NULL` | YES | Soft-delete timestamp. When a company is deleted, this column is set to `now()` instead of removing the row. The company remains in the database for 30 days, allowing recovery. After 30 days, a cleanup process permanently deletes the row and all cascaded data. See the Soft-Delete section below. |
| `created_at` | `TIMESTAMPTZ` | `now()` | NOT NULL | Timestamp of when the company was created. Set once during onboarding and never modified. |
| `updated_at` | `TIMESTAMPTZ` | `now()` | NOT NULL | Timestamp of the last modification to the company record. Updated automatically by application code on any change to name, website, description, logo, etc. |

### Indexes on `companies`

| Index Name | Columns | Type | Condition | Purpose |
|------------|---------|------|-----------|---------|
| `companies_pkey` | `id` | B-tree (PK) | — | Primary key lookup |
| `idx_companies_deleted_at` | `deleted_at` | B-tree (partial) | `WHERE deleted_at IS NULL` | Fast filtering of active (non-deleted) companies. This partial index ensures that the vast majority of queries -- which only care about active companies -- can quickly skip soft-deleted rows without scanning the full table. |

### RLS Policies on `companies`

Row Level Security is enabled on the `companies` table. The following policies control access:

| Policy Name | Operation | USING / WITH CHECK | Description |
|-------------|-----------|-------------------|-------------|
| `companies_select` | SELECT | `id IN (SELECT company_id FROM public.users WHERE id = (select auth.uid())) AND (deleted_at IS NULL OR EXISTS (SELECT 1 FROM public.users u WHERE u.id = auth.uid() AND u.role IN ('admin', 'owner')))` | Users can only view their own company. Soft-deleted companies are hidden from regular members but remain visible to `admin` and `owner` roles for recovery purposes. |
| `companies_insert` | INSERT | `(select auth.uid()) IS NOT NULL` | Any authenticated user can create a company. This is used during the onboarding flow when a new user creates their organization. The policy is intentionally permissive on INSERT because the onboarding logic handles validation. |
| `companies_update` | UPDATE | `id IN (SELECT company_id FROM public.users WHERE id = (select auth.uid()))` | Users can only update their own company's record. This allows changing name, website, logo, description, and industry. |
| `companies_service` | ALL | `(select auth.role()) = 'service_role'` | The Supabase service role (used by API routes with `createServerSupabaseClient()`) has full access to all companies. This is necessary for admin operations, Stripe webhooks, and cross-company queries in the [[Command Center]]. |

### Soft-Delete Mechanism

The `companies` table implements soft-delete via the `deleted_at` column, introduced in migration `20260323000002_production_audit_fixes.sql`. When a company owner requests account deletion:

1. The `deleted_at` column is set to `now()` instead of running `DELETE`.
2. The partial index `idx_companies_deleted_at WHERE deleted_at IS NULL` ensures that normal queries efficiently skip deleted companies.
3. The RLS policy `companies_select` excludes soft-deleted companies from results for regular `member` users. Only `admin` and `owner` roles can still see soft-deleted companies, allowing them to initiate recovery within the 30-day window.
4. All child entities (contacts, campaigns, calls, etc.) remain intact via their CASCADE foreign keys -- they are not deleted until the company row is permanently removed.
5. After 30 days, a background cleanup process (triggered from the admin [[Command Center]]) permanently deletes the company row, which cascades to all child tables.

This approach prevents accidental permanent data loss and gives customers a grace period to recover their account.

## Database Table: `company_settings`

The `company_settings` table stores operational configuration for each company. It has a strict 1:1 relationship with `companies` -- every company has exactly one settings row, created during [[Onboarding Flow|onboarding]].

| Column | Type | Default | Nullable | Description |
|--------|------|---------|----------|-------------|
| `company_id` | `UUID` | — | NOT NULL | Primary key and foreign key to `companies(id)` with `ON DELETE CASCADE`. This dual role enforces the 1:1 relationship: there can only be one settings row per company, and deleting the company automatically deletes its settings. |
| `default_voice` | `TEXT` | `'maya'` | YES | The default Bland AI voice used for new agents created by this company. Defaults to "Maya" (Young American Female, voice ID `2f9fdbc7-4bf2-4792-8a18-21ce3c93978f`). Can be changed in the agent configuration modal. See [[Voice Catalog]] for all 51 available voices. |
| `default_interval_minutes` | `INTEGER` | `5` | YES | The default interval in minutes between consecutive calls in a campaign. Controls the pacing of outbound calling. Can be overridden per-campaign in the campaign settings. |
| `default_max_duration` | `INTEGER` | `5` | YES | The default maximum duration in minutes for a single call. Plan-enforced limits may override this value (e.g., Free plan caps at 3 minutes). See [[Plan Features]] for per-plan limits. |
| `test_phone_number` | `TEXT` | — | YES | A phone number used for test calls. When a user clicks "Test Call" in the agent configuration modal, the system dials this number. Stored in E.164 format (e.g., `+14155551234`). |
| `bland_api_key` | `TEXT` | — | YES | Reference to the Bland AI API key. In the current master-key architecture (V4), all companies share a single master Bland API key stored in the `BLAND_API_KEY` environment variable. This field exists for historical reasons and for potential future per-company sub-account support. |
| `openai_api_key` | `TEXT` | — | YES | Optional per-company OpenAI API key. If set, the company's calls will use this key for post-call AI analysis instead of the platform's shared key. Encrypted using AES-256-GCM via `encryptToken()` from `src/lib/encryption.ts`. |
| `settings` | `JSONB` | `'{}'` | YES | A flexible JSONB column for additional settings that do not warrant dedicated columns. Currently stores preferences like notification settings, default campaign configurations, and UI preferences. |
| `bland_subaccount_id` | `TEXT` | — | YES | Identifies the Bland AI sub-account for this company. In the current V4 architecture, this is always set to `'master'` for the admin account. The master-key architecture means all companies share the same Bland AI account, with isolation handled at the Callengo application layer via `company_id`. See [[Bland AI]] for the full architecture. |
| `created_at` | `TIMESTAMPTZ` | `now()` | NOT NULL | Timestamp of when the settings row was created. Always matches the company creation time since both are created in the same onboarding transaction. |
| `updated_at` | `TIMESTAMPTZ` | `now()` | NOT NULL | Timestamp of the last modification to any settings field. Updated by application code. |

### RLS Policies on `company_settings`

| Policy Name | Operation | USING / WITH CHECK | Description |
|-------------|-----------|-------------------|-------------|
| `company_settings_all` | ALL (SELECT, INSERT, UPDATE, DELETE) | `company_id IN (SELECT company_id FROM public.users WHERE id = (select auth.uid()))` | Company members can read and modify their own company's settings. Any team member can change settings, not just owners/admins -- this is intentional to allow members to update test phone numbers and default voices. |
| `company_settings_service` | ALL | `(select auth.role()) = 'service_role'` | Service role has full access for API operations, admin panel, and webhook processing. |

## Key Business Rules

1. **One company per user.** Every [[User]] belongs to exactly one Company. There is no mechanism for a user to belong to multiple companies simultaneously. If a user needs access to a different company, they must create a separate account with a different email address.

2. **Tenant isolation via RLS.** All data access is filtered by `company_id` through PostgreSQL Row Level Security policies. Even if application code has a bug that omits a `WHERE company_id = ...` clause, the RLS policies prevent cross-tenant data leakage at the database level. This is the single most critical security boundary in the system.

3. **Cascading deletes.** All child tables use `ON DELETE CASCADE` on their `company_id` foreign keys. When a company is permanently deleted (after the 30-day soft-delete window), all contacts, agents, campaigns, calls, subscriptions, add-ons, integrations, and every other piece of data is automatically removed by PostgreSQL.

4. **Settings created during onboarding.** The `company_settings` row is created in the same transaction as the `companies` row during the [[Onboarding Flow]]. Application code can safely assume that every company has a corresponding settings row.

5. **Context extraction for AI agents.** The `context_data`, `context_summary`, and `context_extracted_at` columns support an automated workflow where Callengo scrapes the company's `website` URL, extracts structured information, and generates a natural-language summary using GPT-4o-mini. This summary is injected into agent system prompts so that AI agents can speak knowledgeably about the company's products, services, and value propositions during calls.

6. **Bland AI master-key architecture.** The `bland_api_key` and `bland_subaccount_id` columns in `company_settings` reflect the V4 architecture where all companies share a single Bland AI master API key. Tenant isolation for calls is handled entirely at the Callengo application layer using `company_id` in call metadata. The `bland_subaccount_id` is set to `'master'` for the admin account. See [[Bland AI]] for the full architecture details.

7. **Admin visibility.** The [[Command Center]] (`/admin/command-center`) provides platform owners with a cross-company view. The admin API endpoints use the `service_role` to bypass company-scoped RLS and aggregate data across all tenants. The Clients tab shows per-company usage, profit, Bland AI cost, and active add-ons.

## Lifecycle

### Creation

1. User signs up via `/auth/signup` (email/password or OAuth: Google, GitHub).
2. Supabase Auth creates a row in `auth.users`.
3. The onboarding flow at `/onboarding` prompts for company name, website, and industry.
4. A transaction creates:
   - A row in `companies` with the provided name, website, and industry.
   - A row in `company_settings` with default values (voice: maya, interval: 5 min, max duration: 5 min).
   - A row in `users` linking the auth user to the new company with `role = 'owner'`.
   - A row in `company_subscriptions` with the Free plan.
5. The user is redirected to `/home`.

### Modification

- Company name, website, logo, description, and industry can be updated via the Settings page (`/settings`).
- Company settings (default voice, interval, max duration, test phone) can be updated via the Settings page or the Agent Configuration Modal.
- Context data is extracted/refreshed on demand or automatically when the website changes.

### Deletion (Soft-Delete)

1. The company owner initiates account deletion from Settings.
2. `deleted_at` is set to `now()`.
3. The company becomes invisible to regular members via RLS.
4. Admin/owner can still see and recover the company for 30 days.
5. After 30 days, the admin cleanup process permanently deletes the row, cascading to all child tables.
